System Design

1. High-Level Architecture

The platform is structured as two independently deployable processes — a Next.js frontend and a FastAPI backend — communicating exclusively over an HTTP REST API with a single Server-Sent Events (SSE) channel for streaming chat responses. This separation enforces a strict client/server boundary: all business logic, model access, and persistence reside in the backend; the frontend is a pure rendering and interaction layer with no direct database or LLM access.

External dependencies are confined to two integration points: the OpenAI API (used by the platform model provider and the AI analysis sub-system) and the Garak CLI (invoked as a managed subprocess for red-teaming framework runs). Both integrations are behind abstract interfaces in the domain layer, so neither affects the core business logic.

Deployment topology: both processes run on the same host during development and are configured via environment variables (OPENAI_API_KEY, DATABASE_URL, NEXT_PUBLIC_API_URL). No separate message broker, cache, or authentication service is required.

2. Design Principles & Architectural Patterns

Four architectural patterns govern every structural decision in the backend codebase. Their combined effect is a system in which the domain model is entirely free of framework and I/O concerns, and in which any provider, runner, or persistence technology can be replaced without touching business logic.

Clean / Layered Architecture

The backend is organised into four explicit layers — API, Application, Domain, and Infrastructure — with dependencies pointing strictly inward. The domain layer defines only pure Python dataclasses and abstract base classes; it has no imports from FastAPI, SQLAlchemy, or any third-party library. Outer layers depend on inner ones; inner layers never depend on outer ones.

Repository Pattern

All persistence is mediated through a TestRepository class that translates between domain objects (Test, ModelSpec, RunnerSpec) and SQLAlchemy ORM models. Application services receive an injected SQLAlchemy Session and call repository methods; they are never aware of the underlying SQL dialect or table schema.

Strategy Pattern via Abstract Base Classes

Three families of behaviour are defined as abstract base classes in the domain layer: ModelProvider (how to call a model), TestRunner (how to execute a test), and Mitigation (how to filter or modify a prompt/response). Concrete implementations are registered in module-level registries (MODEL_REGISTRY, MITIGATION_REGISTRY) and resolved at runtime by ProviderRouter and RunnerRouter.

Dependency Injection via Container

A central Container class (instantiated once at startup in main.py) constructs and wires all service objects. FastAPI route handlers receive services via Depends(), ensuring that no service creates its own dependencies. This makes the entire service graph testable in isolation by substituting mock objects at the container boundary.

3. Frontend Architecture

The frontend is a Next.js 15 application using the App Router with React 19 Server and Client Components. It is organised around seven top-level pages, each backed by a dedicated TypeScript module under app/pages/. All LLM-facing and persistence operations are delegated to the backend; the frontend holds no persistent state beyond what is managed in React component state and the browser session.

The types/testing.ts module is the interface contract between frontend and backend. Every REST response is typed against the definitions in this file, providing compile-time safety for field access and ensuring that any backend schema change propagates as a TypeScript type error rather than a runtime failure.

Model and Mitigation Registries

The infrastructure layer exposes two module-level dictionaries that act as service locators for pluggable strategies. MODEL_REGISTRY maps model identifiers (e.g. "gpt-5.2") to ModelProvider instances; MITIGATION_REGISTRY maps slug strings (e.g. "pre_filter") to Mitigation instances. Both registries are populated at module import time, making new providers and mitigations addable without modifying any service code.

5. AI Analysis Pipeline

Every prompt test run produces a MitigationAnalysis domain object that reports whether the model's response was compromised by the injected prompt. This judgment is produced by a two-stage hybrid analyser in the infrastructure layer: a deterministic rule engine (weighted 60%) combined with an AI semantic classifier (weighted 40%). The final risk score is computed using a probabilistic combination formula, score = 1 − ∏(1 − wᵢ · sᵢ), which keeps the result naturally bounded in [0, 1].

6. End-to-End Workflow

The sequence diagram below traces a complete prompt-runner test execution from the moment the user initiates a run in the browser to the moment results are returned. The Garak framework path diverges after runner dispatch and is not shown in full.

sequenceDiagram participant FE as Frontend participant API as API Layer participant SVC as TestService participant RR as RunnerRouter participant PR as PromptRunner participant MIT as Mitigation chain participant MP as ModelProvider participant AZ as PromptTestAnalyser participant DB as TestRepository FE->>API: POST /tests/{id}/run API->>SVC: run_test(test_id, session) SVC->>DB: get_by_id(test_id) DB-->>SVC: Test domain object SVC->>RR: get_runner(runner_type) RR-->>SVC: PromptRunner instance SVC->>PR: run(test) PR->>MIT: apply_pre_input(prompt) MIT-->>PR: filtered / augmented prompt PR->>MP: complete(messages) MP-->>PR: model response text PR->>MIT: apply_post_output(response) MIT-->>PR: filtered / augmented response PR->>AZ: analyse(response, prompt, context, mitigation_results) AZ-->>PR: MitigationAnalysis (flagged, score, reason) PR-->>SVC: RunResult SVC-->>API: RunResult API-->>FE: JSON RunResult

Mitigation objects in the chain are resolved from MITIGATION_REGISTRY using the slug list stored in EnvironmentSpec.mitigations. Each mitigation implements exactly one of the MitigationLayer stages (PRE_INPUT, POST_OUTPUT, PROMPT, INFRASTRUCTURE, MONITORING), and the runner iterates them in layer order. Mitigation verdicts are aggregated alongside the analyser result to produce the composite MitigationAnalysis.

7. Database Architecture

Persistence is provided by a single SQLite file accessed through SQLAlchemy's ORM. The schema comprises three tables. Complex nested objects (ModelSpec, EnvironmentSpec, RunnerSpec) are stored as JSON in TEXT columns and deserialised by TestRepository._to_domain() on read. This avoids the overhead of normalised sub-tables while keeping the domain model free of ORM annotations.

erDiagram tests { TEXT id PK TEXT name TEXT model_json TEXT environment_json TEXT runner_json TEXT created_at } test_configurations { TEXT id PK TEXT name TEXT model_json TEXT environment_json TEXT runner_json TEXT created_at } prompt_enhancements { TEXT id PK TEXT original_prompt TEXT enhanced_prompt TEXT enhancement_type TEXT created_at }

JSON column strategy: The model_json, environment_json, and runner_json columns each store a complete serialisation of their corresponding domain dataclass (produced via Python's dataclasses.asdict()). On read, TestRepository reconstructs the domain objects field-by-field, applying type coercions (e.g. converting runner["type"] to a RunnerType enum). This design allows the domain model to evolve independently of the SQL schema, with migration handled by adding new optional keys to the JSON rather than altering table columns.

The test_configurations table stores saved (but not yet executed) test templates with the same JSON column structure as tests. Separating executed tests from saved configurations allows the UI to present a clean distinction between "saved test presets" and "historical run results".

8. API Design

The backend exposes a RESTful HTTP API mounted under /api/v1 with six route modules. All request and response bodies are JSON validated by Pydantic v2 schemas. The API contract is formally documented by FastAPI's auto-generated OpenAPI specification, accessible at /docs during development.

Request / Response Contract

The POST /tests/{id}/run endpoint is the most structurally complex, returning a RunResult object that varies by runner type:

FieldTypePresent forDescription
output string All Human-readable summary of the run outcome
analysis TestAnalysis All Composite verdict: flagged (bool), score (0–1), reason
started_at / finished_at ISO-8601 string All Wall-clock timestamps for the run
attempts AttemptResult[] Framework runs Per-probe attempt details: prompt sent, output received, blocked, compromised, HTTP statuses
report_html_url string | null Framework runs Relative URL to the Garak-generated HTML report file, served statically

The SSE endpoint at POST /chat/stream emits newline-delimited data: events. Each event carries a JSON object with a single token field. A terminal data: [DONE] event signals end-of-stream, mirroring the convention used by the OpenAI streaming API.
