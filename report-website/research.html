Research and Technical Overview

Prompt Injection Research, Platform Research, and Technical Decisions

This page is split into two parts: the prompt injection research that informed the project, and the platform and technical decisions evidenced by the implemented codebase.

Part 1 - Prompt Injection Research

1. Introduction and Background

Since the public release of ChatGPT in late 2022, LLMs have moved rapidly from research labs into mainstream products, internal tooling, and enterprise workflows. As these systems become embedded in decision-making, content generation, retrieval, and tool execution, their security properties matter just as much as their usefulness.

Prompt injection is a particularly important risk because it exploits how language models fundamentally work. Unlike traditional software, LLMs process trusted instructions and untrusted content through the same natural-language channel. This creates ambiguity between “instructions” and “data”, allowing attackers to manipulate behaviour through direct prompts, retrieved documents, tool outputs, or even multimodal inputs.

This matters even more in enterprise environments. Once a model is connected to retrieval pipelines, APIs, internal documents, or tool-calling systems, prompt injection is no longer just a chatbot failure: it becomes a route to data leakage, unsafe automation, and loss of control over downstream actions.

Key insight: prompt injection is not simply a UI problem or a badly written prompt. It is a structural security problem created by the way LLMs combine instructions, context, external data, and tool access into one reasoning flow.

2. Research Report

To ground the project in evidence rather than assumptions, we produced a dedicated research report on prompt injection protection. This report became the basis of our internal knowledge base and informed how we framed the problem, compared existing approaches, and prioritised the features and evaluation criteria of the platform.

What the research covered

  • Direct and indirect prompt injection
  • Instruction-data boundary confusion
  • Obfuscation and encoding-based attacks
  • Tool-use, MCP, and workflow contamination risks
  • Mitigations used in research and industry
  • Testing pipelines, frameworks, and evaluation metrics

How we used it

  • To define the project scope around realistic enterprise threats
  • To identify which attack surfaces matter beyond simple jailbreaks
  • To shape the evaluation strategy and testing priorities
  • To compare defensive approaches instead of relying on one single safeguard
  • To understand where current public tools are useful and where gaps remain

3. Vulnerability Taxonomy

Prompt injection is not one single attack pattern. It is a family of vulnerabilities caused by how LLM systems combine natural-language instructions, external context, memory, and tool access. Our taxonomy groups the main classes of risk that appear repeatedly in literature and real deployment scenarios.

⚡ Direct Prompt Injection

A user explicitly attempts to override intended behaviour with adversarial instructions, for example by asking the model to ignore system guidance, reveal hidden instructions, or behave outside policy.

User Input

📨 Indirect Prompt Injection

Malicious instructions are hidden inside external content such as emails, webpages, database entries, or documents that the model later reads and mistakenly treats as actionable.

External Data

🔀 Instruction-Data Boundary Confusion

Because system prompts, user input, retrieved text, and tool outputs are processed in a shared text context, the model may fail to maintain a strict distinction between “content” and “instructions”.

Architecture

🎭 Obfuscation and Encoding Attacks

Attackers may disguise malicious instructions through Base64, Unicode homoglyphs, multilingual phrasing, emoji patterns, or fragmented prompts to bypass simple filters.

Evasion

🔧 Tool-Use and Action Injection

If an LLM can call tools or APIs, an injection attack may push it to generate unsafe arguments, perform unintended actions, or trust poisoned tool outputs.

MCP / Tools

💾 Memory and Long-Horizon Vulnerabilities

Seemingly harmless prompts can accumulate across multiple turns, gradually steering the model’s behaviour and causing safety drift over time.

Long-Horizon

🖼️ Multimodal Injection

Hidden instructions can be embedded in images or other non-text modalities, creating attack paths that bypass text-only validation or filtering strategies.

Cross-Modal

🌐 Multi-Agent / Workflow Contamination

In agentic systems, one compromised step can contaminate shared memory, planning outputs, or downstream agents, turning a local failure into a broader workflow failure.

Multi-Agent

4. Mitigation Strategies

Our research consistently showed that there is no single fix for prompt injection. Effective protection requires defence in depth: reducing exposure, checking behaviour, constraining actions, and continuously testing the system as it evolves.

Defence in depth: the goal is not to assume that one layer will stop every attack, but to combine preventative, detective, and containment-focused safeguards.

🛡️

Input Validation & Sanitisation

Treating all user input and external content as untrusted, applying filtering, validation, schema checks, or auxiliary classification before the model sees it.

📝

Robust Prompt Design & Context Separation

Structuring prompts so trusted instructions are clearly separated from untrusted data, rather than concatenating everything into one ambiguous block.

🔍

Output Filtering, Guardrails & Oversight

Validating outputs before they are shown to users or used in downstream actions, and adding secondary checks or human review for high-risk operations.

🤖

Multi-Agent Defence Pipelines

Splitting responsibilities across specialised stages such as review, sanitisation, execution, and output checking to reduce single-point compromise.

🧹

Context Sanitisation for Long or External Inputs

Analysing retrieved or long-context content before it enters the prompt, especially in RAG systems, browsers, and pipelines that consume untrusted documents.

🔒

Sandboxing & Permission Control

Restricting what tools the model can call, validating arguments, and requiring approval for sensitive actions so successful injections have limited real-world impact.

⚙️

Model-Level Alignment

Alignment, adversarial fine-tuning, and policy training improve baseline robustness, but should be treated as one layer of defence rather than a full solution.

📊

Monitoring and Governance

Mature deployments require ongoing monitoring, incident response, and continuous evaluation because prompt injection threats evolve over time.

5. Testing Frameworks and Methodologies

The report surveys a broad testing ecosystem, including attacker-model testing, black-box and white-box methods, external-data injection testing, MCP-focused scenarios, and frameworks such as Garak, PromptMe, PromptBench, OpenAI Evals, AgentHarm, HarmBench, and PyRIT. However, the implemented platform does not wire all of those frameworks directly into the product. This section therefore focuses on what the current platform actually uses.

Implemented testing in the platform: the codebase currently supports two practical testing modes: custom prompt testing through the main prompt runner, and framework-based scanning through a framework runner that delegates to Garak.

Prompt Runner

The main testing path in the platform is the prompt runner. Users create a test, choose a model, optionally attach an environment with mitigations, and then run the prompt against either a platform model or an external model endpoint.

For platform models, the backend applies pre-input mitigations, compiles prompt-layer mitigation instructions into the system prompt, applies post-output filtering, and then performs monitoring and analysis after the response is returned.

Framework Runner

The framework mode currently routes to Garak. The frontend testing page exposes Garak probe selection, and the backend framework runner delegates directly to a Garak-based runner that executes scans, stores results, and exposes saved run summaries.

This makes Garak the only framework-backed testing tool that is clearly active in the current platform implementation.

Automated Risk Analysis

After prompt runs, the backend performs a second-stage analysis to determine whether the model response appears to reflect successful prompt injection. This analysis returns a flag, a score, and a reason, giving the user an interpretable risk summary rather than just raw model output.

This is one of the platform’s most important testing choices because it turns raw outputs into an evaluative workflow.

Conversation-Aware Testing

The testing system stores conversation context so that follow-up messages in the same test can keep history. This matters because prompt injection is often not a single-turn problem: the platform can test behaviour in a more realistic conversation setting rather than treating each input as isolated.

External Model Validation

The backend also supports testing external models via user-provided endpoints and response mappings. In that mode, the model can be evaluated as-is, which is useful for comparing external behaviour without forcing the platform’s internal mitigation pipeline onto it.

What is present in the repository but not actively wired as a platform framework

The repository includes a PyRIT runner file, but it is not registered in the active runner registry. The report discusses PromptMe, PromptBench, OpenAI Evals, AgentHarm, AdvBench, HarmBench, and PyRIT as important research frameworks, but those are not exposed as first-class platform testing modes in the current implementation.

What this means for the platform

  • The platform supports both interactive prompt testing and framework-driven scans
  • Garak is the currently implemented framework-level scanner
  • The platform combines raw test execution with post-run risk analysis
  • Conversation context and external model validation make the testing workflow more realistic
  • The broader testing frameworks from the report remain part of the research basis and future expansion space, not the full implemented testing surface

6. Key Findings

The strongest conclusion from our research is that prompt injection should not be treated as a niche jailbreak problem. It is a broad system-level risk that grows as models become more connected, more agentic, and more able to act on the world through tools and workflows.

✅

Direct attacks are only one part of the problem

Simple direct injections matter, but the more difficult and more realistic risks often come from indirect, hidden, or tool-mediated attacks.

⚠️

Obfuscation remains a major weakness

Encoded, multilingual, or otherwise disguised instructions are still much harder to defend against than plain-text attacks.

🔗

Tool and MCP access expand the attack surface

Once models can retrieve content, call APIs, or trigger downstream actions, prompt injection can lead to much more serious consequences than unsafe text generation alone.

🛡️

No single mitigation is enough

Effective protection depends on layered controls: input handling, prompt structure, guardrails, permissions, testing, and monitoring all matter.

📊

Security testing must be continuous

Prompt injection resilience cannot be “proved” once and then assumed forever. Testing needs to continue as prompts, tools, models, and integrations change.

🏗️

This is mainly an architectural and operational challenge

Safer deployment depends less on choosing a “perfect” model and more on how the system is structured, constrained, evaluated, and governed in practice.

These findings directly informed the platform direction: focus on practical threat visibility, evaluation support, and deployment-aware safety thinking rather than pretending prompt injection can be eliminated with a single prompt or model choice.

Part 2 - Platform Research and Technical Overview

7. Existing Solutions (and Where We Took Inspiration)

Before deciding what our platform should do, we reviewed both practical security tooling and broader enterprise-facing AI platforms. Our goal was not to copy an existing product, but to understand how current solutions approach prompt injection, red-teaming, evaluation, and operational safety.

Foundry

Foundry was useful as a high-level enterprise reference point. It helped frame the idea of working with multiple models and managed providers rather than designing the product around a single hard-coded model family.

NVIDIA Garak

Garak was the most directly reflected external testing framework in the implemented platform. It demonstrated the value of repeatable adversarial probing and informed the framework-scan capability exposed through the Testing page.

Microsoft PyRIT

PyRIT was relevant from a research and architecture perspective because it frames prompt injection testing as structured red-teaming. The repository includes a PyRIT runner file, which shows it was considered in the codebase, but it is not part of the currently active runner registry.

PromptBench

PromptBench informed the importance of adversarial benchmarking and comparing behaviour across prompt variants rather than relying on one or two examples.

OWASP PromptMe

PromptMe was useful because it connects testing to recognised OWASP threat scenarios and reinforces a developer-facing, application-security view of prompt injection.

OpenAI Evals and benchmark suites

OpenAI Evals, together with benchmark-oriented resources such as AgentHarm, AdvBench, and HarmBench, influenced the idea that testing should be reproducible, extensible, and suitable for regression workflows rather than one-off manual checks.

What we took inspiration from

  • Framework-assisted testing rather than only manual demonstrations
  • Support for multiple models and providers instead of a single locked stack
  • Security evaluation as a workflow, not a one-time benchmark
  • Developer-facing outputs that explain risk, not just raw model responses
  • An enterprise framing focused on governance, repeatability, and deployment realism

8. Frontend

Important note: the repository clearly documents the implemented frontend stack, but it does not include a formal written shortlist of rejected frontend frameworks. For that reason, this section records the final frontend choices and the trade-offs evidenced by the codebase, rather than inventing a comparison against tools that are not documented.

The implemented frontend is a React single-page application built with TypeScript and bundled with Vite. Styling is based on Tailwind CSS v4, with shadcn/ui primitives built on Radix UI. Routing is handled by react-router-dom, theming by next-themes, animations by motion, and PDF export by @react-pdf/renderer.

React + TypeScript

Why it fits this project: the platform is not a single report page but a multi-view application with testing tools, a vulnerability catalogue, mitigations, a prompt enhancer, and a security assistant. A component-based React SPA makes that easier to organise and extend.

Trade-off: a SPA introduces more frontend structure and state management complexity than a static site, but it better supports interactive workflows.

Vite

Why it fits this project: Vite supports fast local development and straightforward builds, which is useful in a project with frequent frontend iteration.

Trade-off: Vite is a build tool rather than a full application framework, so some architectural decisions remain the team’s responsibility.

Tailwind CSS v4

Why it fits this project: Tailwind makes it easier to maintain visual consistency across many pages and components while supporting rapid UI iteration.

Trade-off: utility-heavy markup can become verbose, but it keeps styling close to the component where it is used.

shadcn/ui + Radix UI

Why it fits this project: this combination gives accessible, composable UI primitives without forcing the whole design into a rigid visual system. That fits a coursework platform that needs both polish and flexibility.

Trade-off: it provides primitives rather than full page design out of the box, so more composition work is needed. The final repository also explicitly removes unused MUI/Emotion dependencies, which shows the implemented component direction is Radix/shadcn rather than Material UI.

react-router-dom

Why it fits this project: the platform has multiple routes such as Home, Mitigations, Testing, Prompt Enhancer, and Security Knowledge Assistant, so explicit client-side routing is a natural fit.

Trade-off: client-side routing requires frontend route management, but it keeps navigation smooth inside the SPA.

next-themes, motion, and @react-pdf/renderer

Why they fit this project: next-themes supports the repository rule that dark mode is mandatory, motion supports page transitions and interactive polish, and @react-pdf/renderer enables export of vulnerability, mitigation, and test reports from inside the app.

Trade-off: each library adds dependency weight, but each one supports a clearly implemented feature in the shipped frontend.

9. Backend

Important note: the repository clearly documents the implemented backend stack, but it does not include a formal written shortlist of rejected backend frameworks. This section therefore focuses on the implemented tools and the trade-offs evidenced by the codebase.

The implemented backend is built in Python using FastAPI. It uses SQLite for persistence, SQLAlchemy and SQLModel-related pieces for database interaction, httpx for outbound HTTP model calls, and a provider-routing layer to abstract OpenAI, Anthropic, and OpenAI-compatible model endpoints.

Python

Why it fits this project: Python is a natural fit for AI integration, test tooling, backend orchestration, and security-oriented prototyping. It also aligns well with the libraries used for HTTP clients, model integration, and backend testing.

Trade-off: Python is not the most performance-focused option for all workloads, but the current platform prioritises flexibility, integration, and iteration speed.

FastAPI

Why it fits this project: FastAPI gives a clean API layer for test creation, model execution, chat streaming, prompt enhancement, and model listing. It also suits typed request/response schemas and rapid API development.

Trade-off: a separate backend adds deployment complexity compared with a purely static frontend, but it is necessary here because the platform performs model calls, persistence, analysis, and streaming.

SQLite

Why it fits this project: SQLite is simple and lightweight, which suits a coursework platform that needs to persist tests, framework runs, and prompt enhancements without requiring external database setup.

Trade-off: SQLite is less scalable than a full client-server database, but it is sufficient for the implemented platform and reduces operational overhead.

SQLAlchemy / SQLModel-related database layer

Why it fits this project: the backend uses a structured persistence layer with models, repositories, and sessions instead of raw ad hoc file storage. That fits the need to save tests and enhancement results in a maintainable way.

Trade-off: an ORM layer adds abstraction, but it keeps persistence more organised as the backend grows.

httpx and provider implementations

Why it fits this project: the backend needs to call different model providers through HTTP. httpx makes that practical while keeping the provider implementations separate and reusable.

Trade-off: supporting multiple providers requires more configuration and abstraction, but it avoids tying the platform to a single vendor path.

Provider router architecture

Why it fits this project: all model calls go through a router that resolves the chosen model to the correct provider implementation. This keeps routes, services, and runners free from provider-specific logic and makes the system easier to extend with more models later.

Trade-off: abstraction adds architectural complexity, but it directly supports the platform’s multi-provider research goals.

10. Technical Decisions

Beyond the choice of tools, the repository shows several important architectural decisions that shape how the platform works. These decisions are clearer and more trustworthy than trying to guess undocumented “alternatives”.

React SPA + FastAPI backend

The project is intentionally split into a React single-page frontend and a FastAPI backend. This separates UI concerns from model routing, test orchestration, persistence, and analysis.

REST + SSE rather than polling for chat

The security assistant uses server-sent events for streaming chat responses. This makes the assistant feel more interactive and better matches the token-by-token behaviour users expect from AI systems.

Provider abstraction instead of provider-specific logic everywhere

All model calls are routed through the provider router. This avoids scattering provider-specific request logic throughout routes and services and makes the backend easier to maintain.

Layered backend architecture

The backend is explicitly divided into API, application, domain, and infrastructure layers. This keeps HTTP, business logic, domain logic, and external integrations separate instead of mixing everything together.

Prompt enhancer with verification

The prompt enhancer is not a single rewrite step. It is a three-stage pipeline: improve, prepend mitigation guidance, and verify. The verification stage exists so the platform does not blindly trust its own enhancement output.

Knowledge-base-backed assistant instead of raw chat only

The assistant first searches an internal knowledge base, augments the query with matched vulnerabilities and mitigations, and then calls the model. This is a deliberate choice to ground responses in platform knowledge rather than rely only on raw generation.

localStorage instead of sessionStorage for chat persistence

The repository explicitly records that chat history was moved to localStorage so conversations survive tab closure and browser restart. That is a direct implemented decision, not a guess.

Framework scans currently standardised on Garak

Although the broader report discusses multiple evaluation frameworks, the active framework runner currently standardises on Garak. This keeps the implemented framework path narrow and maintainable while still supporting a serious scan workflow.

Testing external models as-is

For external model validation, the runner skips the platform’s internal mitigation flow and tests the endpoint as provided. This is a useful decision because it allows comparison of external systems without changing their behaviour first.

Dark mode as a non-negotiable UI rule

The repository rules explicitly state that dark mode support is mandatory. This is not just visual polish: it is a design-system decision that affects every component and page in the frontend.
