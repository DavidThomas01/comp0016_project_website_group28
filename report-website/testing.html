Testing strategy

The app repo does not ship a formal testing strategy doc or a line-by-line link to MoSCoW — that lives on the requirements side of this report. In practice, effort went into backend correctness so the flows in the README stay believable: catalogue and content, the Testing API (POST /api/tests, runs, analysis), Security Assistant chat (SSE), Prompt enhancer, and Garak-style scans via GarakRunner calling the external garak CLI.

What we lean on

  • Automated: pytest in backend/ (pytest-asyncio for async code).
  • Optional live calls: integration tests need API keys and --run-integration (see below).
  • Frontend: no test script in frontend/package.json — only dev and build.
  • CI: no GitHub Actions (or similar) found in the repo.

How to run the backend suite

  • cd backend then pip install -e ".[dev]"
  • PYTHONPATH=src pytest (see backend/README.md)
  • Integration tests are skipped by default. Use pytest --run-integration — note backend/README.md mentions pytest -m integration, which does not match how conftest.py gates things.

Unit and integration testing

Unit tests

Why: Catch regressions without calling real LLMs most of the time — models, prompt compilation, mitigations, routes, SQLite repos, Garak/prompt runners, analysers.

Tools: pytest ≥ 8, pytest-asyncio ≥ 0.23 (local run resolved to pytest 9.0.2, pytest-asyncio 1.3.0). Config in backend/pyproject.toml and backend/pytest.ini.

Outcome: Large backend surface covered; no frontend unit pipeline. Pytest may warn about test classes named Test / TestResult in src/ — noisy but harmless.

Integration tests

Why: Prove real HTTP calls to OpenAI, Anthropic, or OpenAI-compatible endpoints when keys exist.

Where: e.g. backend/tests/integration/models/ (live OpenAI, compatible, Anthropic modules).

How: pytest --run-integration plus env vars for keys (FOUNDRY_GPT52_KEY, etc. per README). Many tests skip if a key is missing.

Outcome: Supported in code, opt-in, key-dependent, not run in CI because there is no CI.

Prompt testing

This is where the product logic meets automated checks: risk scores, “flagged” lines, and Garak wrapping.

Rules in code

  • Prompt test path: PromptTestAnalyser treats output as flagged when risk_score ≥ 0.4 (_FLAGGED_THRESHOLD in prompt_test_analyser.py).
  • Garak: GarakRunner runs the garak binary; DEFCON is mapped to a 0–1 score; flagged if score > 0 (with a special case when Azure content filtering blocks everything).
  • PyRIT: pyrit_runner.py is a stub returning a canned JSON shape — not a real PyRIT harness.
  • Tests: pytest covers analysers, scoring, Garak runner (mocked subprocess). Some cases were among the five failures — worth fixing before claiming a clean bill of health.

Garak run outputs are not checked in (e.g. garak_runs/ gitignored). The Testing UI still lets people kick off framework scans from the front end.
