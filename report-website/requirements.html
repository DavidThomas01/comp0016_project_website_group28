Requirements

Functional and non-functional requirements for the Prompt Injection Protection platform, defined collaboratively with Avanade and prioritised using the MoSCoW method.

Requirements Overview

The requirements were defined to guide the design and implementation of a web-based platform for exploring prompt injection vulnerabilities, mitigations, and testing methodologies. Requirements are categorised as functional and non-functional, and prioritised using the MoSCoW method.

Functional Requirements

✅ Must Have

Home Page

  • The system must provide a home page introducing prompt injection, its associated risks, and a high-level explanation of how it works.
  • The home page must display a list of known prompt injection vulnerabilities.
  • Each vulnerability must be clickable and include a short summary.

In-Depth Vulnerability View

  • Selecting a vulnerability must navigate to an in-depth vulnerability view.
  • The view must include a technical explanation of the vulnerability.
  • A working example demonstrating the vulnerability must be provided.
  • At least one mitigation strategy must be clearly explained.
  • A comparison panel must show model behaviour with and without mitigation.

Knowledge Base

  • The system must provide a knowledge base of prompt injection vulnerabilities.
  • Each entry must include technical explanations and working attack examples.
  • Clear explanations of mitigation techniques must be provided.

Testing View

  • A testing view must be accessible from the main navigation bar.
  • Users must be able to create named custom tests for individual mitigations.
  • Tests must be run through a chat-style interface processing a single prompt.
  • Mitigations must be individually enabled or disabled per test.
  • Tests must return a clear pass/fail result based on security expectations.
  • Test data must be preserved locally using an SQL database.
  • All tests must run on a fixed model (GPT-5.0).

📌 Should Have

Sandbox Input

Allow prompt testing with and without mitigations applied in a sandboxed environment.

Pseudocode Mitigations

Present mitigation implementations in pseudocode for educational clarity.

PDF Export

Export vulnerability views and test results as downloadable PDF reports.

Knowledge Base Chatbot

A chatbot trained on the knowledge base to answer theoretical and practical questions.

💡 Could Have

  • Mitigation examples in multiple programming languages.
  • Internal navigation links suggested by the chatbot.
  • An AI-powered prompt enhancer based on selected vulnerabilities and mitigations.
  • Support for automated testing pipelines using frameworks such as NVIDIA Garak and Microsoft PyRIT.

🚫 Won't Have

No authentication system. No automatic fixing of detected prompt injection attacks. No modification of custom or non-standard models.

Non-Functional Requirements

✅ Must Have

  • Web delivery: The system must be delivered as a web application.
  • Clear interface: The interface must be understandable for technical and semi-technical users.
  • Responsive interactions: The chatbot and testing views must provide acceptable interactive response times.
  • Intuitive navigation: The interface must be logically structured and easy to use.
  • Dark mode: The system must support a dark mode interface.

📌 Should Have

🎨

Avanade Branding

Consistent colour schemes aligned with Avanade visual identity and standardised iconography.

🟢

Status Indicators

Red for vulnerabilities or failing tests; green for mitigations or passing tests.

💡 Could Have

Mapping of vulnerabilities and mitigations to standards such as OWASP Top 10 for LLMs and MITRE ATLAS.

🚫 Won't Have

  • No persistent backend server.
  • No guaranteed compatibility with all LLM providers.
  • No fully custom automated testing pipelines.
