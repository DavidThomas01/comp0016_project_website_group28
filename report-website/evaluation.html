Evaluation

How we assessed the delivered platform against the requirements, the team’s contributions, and a critical look at what worked and what did not.

Summary of Achievements

The detailed tables are in Requirements traceability. This section is a brief summary.

Functional — Must have

Home page, vulnerability catalogue with in-depth views (examples, mitigations, comparison), structured knowledge content, and a full testing workbench.

See a fuller summary

Home

  • Introduces prompt injection, risks, and how attacks work; lists vulnerabilities with short summaries and navigation into detail.

Vulnerability view

  • Technical explanation, behaviour example, mitigation(s), and a comparison of unmitigated vs mitigated model output.

Knowledge base

  • Worked examples, attack demonstrations, and mitigation explanations with enough depth to support learning and testing.

Testing

  • Custom named tests, chat-style runs with per-test mitigations, pass/fail expectations, server-side SQL storage, default platform models in the GPT-5.x family (see evaluation notes for naming vs “GPT 5.0” in the spec).

Non-functional — Must have

Browser-based delivery, clear presentation for technical and semi-technical readers, responsive interactive areas, consistent standards-aligned UI, and dark mode.

See a fuller summary

Delivery & form

  • Web application usable in standard browsers without requiring a persistent backend for basic use.

Clarity & audience

  • Vulnerabilities, mitigations, and test outcomes are explained so both technical and semi-technical users can follow them.

Performance & UI

  • Chat and testing flows stay responsive in normal use; layout and navigation follow the agreed interface patterns.

Preference

  • Dark mode is available as a first-class theme choice.

Critical Evaluation of the Project

A structured assessment of the platform across key dimensions (the report website is evaluated mainly under UI/UX and maintainability). Expand each area for details. Ratings use a simple scale (great, very good, good, adequate) and reflect both strengths and limitations.

User Interface / User Experience

The application is organised around clear journeys: an overview of vulnerabilities, structured educational content, and a testing workbench with chat-style runs and mitigation toggles. We aimed for a layout that works for both technical readers (who want detail) and semi-technical readers (who need plain explanations), consistent with our non-functional requirements. Dark mode is implemented as a first-class preference, which supports longer study sessions and matches expectations for a security-focused tool.

We did not run a large formal usability study with external participants solely for this module; validation came from iterative internal use, structured user-testing sessions organised by the team, and regular feedback from Avanade. That gave us confidence in navigation and clarity, but it is a limitation compared to a dedicated HCI trial with a bigger sample.

Separately, the report website uses a single shared stylesheet, sticky navigation, skip links, and consistent section patterns so documentation stays readable and professional. It is static HTML (no component framework), so repetition across pages is a trade-off we accepted for simplicity and file:// compatibility.

Rating: very good. Strong information architecture and theming on the platform; documentation site is cohesive. Formal UX benchmarking would be needed for a “great” claim.

Functionality

Functionality aligns closely with the MoSCoW specification: a home view that introduces prompt injection and surfaces vulnerabilities; in-depth vulnerability pages with examples and mitigations; a knowledge base with enough depth to support learning; and a testing view with named tests, expectations, and server-side persistence. Optional items such as mapping content to OWASP / MITRE were also delivered where listed in our traceability tables.

The main caveats are inherent to the domain: behaviour depends on hosted language models and APIs, so outputs can vary over time; and a small number of requirements are documented as partial or constrained (for example naming or versioning of default models) where the implementation differs slightly from an original wording. Those are called out in the requirements section rather than hidden.

Rating: great. The team delivered a coherent end-to-end product that meets the agreed scope with transparent handling of edge cases.

Stability

The backend is supported by a substantial automated test suite (pytest), as summarised on the Testing page: hundreds of tests with the majority passing in a standard developer setup. Some tests are skipped or may fail when integration dependencies or environment keys are missing; we document that honestly rather than claiming a perfectly green CI story in every configuration.

On the client side, priority was given to end-to-end exercise of the main flows (browse, learn, run tests) and fixing defects found in those paths. We are not aware of major crashes in normal use at hand-in, but resilience to third-party API outages could be strengthened with more defensive UI messaging everywhere.

Rating: very good. Solid automated coverage on the server and deliberate manual verification of critical journeys; remaining risk is mostly environmental and external services.

Efficiency

SQLite and local-first storage keep data access predictable for coursework scale. The interactive areas (chat-style testing, vulnerability browsing) remain responsive in normal use; the heaviest cost is naturally LLM latency and token use, which we mitigate by using APIs only when the user runs a test or chat action rather than on every navigation.

We did not perform large-scale load testing or cost optimisation for production traffic; Avanade would need caching, rate limits, and hosting sizing if the platform moved beyond a demonstration deployment.

Rating: very good. Appropriate for the intended demonstration and evaluation workload, with a clear path to scale if required.

Compatibility

The product is delivered as a web application for modern browsers, and the report website is plain HTML/CSS/JS so it works when opened locally or via static hosting. The testing workflow depends on configured models and keys; that is a deliberate constraint for reproducibility but means “out of the box” behaviour varies with environment.

We expect the best experience on up-to-date Chromium-based browsers and Safari; Firefox and mobile layouts were not the primary test matrix throughout. MCP-related behaviour can also differ across server implementations, which is a limitation of the ecosystem rather than a single bug in our code.

Rating: good. Broadly compatible with realistic expectations for a student project; edge browsers and MCP diversity remain areas for hardening.

Maintainability

The backend follows a conventional FastAPI structure with tests alongside modules, which should be approachable for Avanade engineers extending APIs or storage. The frontend application is organised by feature areas; the main maintenance cost will be keeping pace with provider APIs and model lifecycles.

The report website prioritises clarity over DRY abstractions: each page is explicit HTML, which is easy to understand but means global changes require careful search-and-replace or disciplined editing. Central CSS and a small shared JS file reduce that burden, and appendices capture deployment and privacy notes for handover.

Rating: good. Maintainable for a small team; the static site and API coupling to vendors are the main long-term maintenance considerations.

Project Management

We used MoSCoW consistently to prioritise must-haves during a fixed academic timeline, held regular contact with Avanade to align scope, and split ownership across backend, frontend, research, and QA so parallel work was possible. Git-based collaboration and issue tracking supported larger changes, though smaller tasks were sometimes coordinated informally.

As with many group projects, workload peaked near deadlines; earlier decomposition of polish items into tracked issues would have reduced last-minute integration pressure. Overall the process delivered a demoable product each term and a documentation set that matches the brief.

Rating: good. Effective partner communication and delivery; room for finer-grained tracking and earlier risk burn-down.

Future work

Promising directions for extending the platform and the research narrative beyond the hand-in scope. These are not commitments of the coursework delivery; they indicate how Avanade or follow-on work could evolve the product.

Multi-model support

Extend the testing framework to multiple LLM providers and model versions so teams can compare behaviour and mitigations side-by-side rather than relying on a single fixed endpoint.

Real-time monitoring

Add production-style monitoring for MCP and tool-using deployments: alerting on suspicious prompt patterns, policy violations, or repeated mitigation failures without blocking all traffic.

Community knowledge base

Open curated contributions for new vulnerability write-ups, mitigations, and test cases with moderation and provenance so the knowledge base stays trustworthy while growing beyond the core team.

Automated discovery

Explore adversarial search and optional attacker-model loops to propose novel injections and tool-abuse sequences, always with human review before results are treated as findings.

Rapid change in LLM capabilities and in MCP adoption means prioritising extensibility, auditability, and partner feedback over chasing every optional feature in a single release.
