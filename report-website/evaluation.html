Summary of Achievements

The detailed tables are in Requirements traceability. This section is a brief summary.

Functional — Must have

Home page, vulnerability catalogue with in-depth views (examples, mitigations, comparison), structured knowledge content, and a full testing workbench.

See a fuller summary

Home

  • Introduces prompt injection, risks, and how attacks work; lists vulnerabilities with short summaries and navigation into detail.

Vulnerability view

  • Technical explanation, behaviour example, mitigation(s), and a comparison of unmitigated vs mitigated model output.

Knowledge base

  • Worked examples, attack demonstrations, and mitigation explanations with enough depth to support learning and testing.

Testing

  • Custom named tests, chat-style runs with per-test mitigations, pass/fail expectations, server-side SQL storage, default platform models in the GPT-5.x family (see evaluation notes for naming vs “GPT 5.0” in the spec).

Non-functional — Must have

Browser-based delivery, clear presentation for technical and semi-technical readers, responsive interactive areas, consistent standards-aligned UI, and dark mode.

See a fuller summary

Delivery & form

  • Web application usable in standard browsers without requiring a persistent backend for basic use.

Clarity & audience

  • Vulnerabilities, mitigations, and test outcomes are explained so both technical and semi-technical users can follow them.

Performance & UI

  • Chat and testing flows stay responsive in normal use; layout and navigation follow the agreed interface patterns.

Preference

  • Dark mode is available as a first-class theme choice.

Known bugs

We tested the core parts of the product carefully and addressed the issues we found. The main user flows have been checked end-to-end, and at this stage we are not aware of any major bugs in the current version. While there were some challenges during development, these were worked through to ensure the key features behave as expected.

We see this as a solid base for Avanade to build on. We have included clear documentation so that anyone can understand the platform, maintain it, and extend it with new functionalities over time. We are also encouraged by the positive feedback from Avanade and Microsoft, as well as the opportunity to share the work through an article on the AI Foundry blog and an upcoming presentation on the AI Foundry Discord.

Critical Evaluation of the Project

A structured assessment of the platform across key dimensions. Expand each area for details. Ratings use a simple scale (great, very good, good, adequate) and reflect both strengths and limitations.

User Interface / User Experience

We built the platform to feel modern and easy to use. We added clear feedback when users interact with the interface, and simple elements that make moving between screens feel smooth and not confusing.

Search, filters, and clear navigation help users find what they need without extra steps. We wanted using the app to stay as simple as possible.

Dark mode works well across the whole product, which was key for our industry partner.

We regularly reviewed the UI with our industry partner, Avanade, to stay aligned with their expectations. We also looked at similar platforms, such as Microsoft AI Foundry, to compare layout and how information is presented. We showed the platform to people outside the project and used their feedback to make small improvements.

The design hides heavy technical detail by default. Users who want a simple overview can use it easily, while those who want more depth can open the technical sections when needed. This helps avoid overwhelming less technical users.

Rating: great. We are very happy with how it feels, and the positive feedback we received from our partner on it.

Functionality

We delivered all of the required features set out in the MoSCoW specification. Only one requirement was not implemented, as it was redundant in the context of the final system and did not add meaningful value. All other core and optional features were completed as planned.

In addition, we extended the platform beyond the original scope. We introduced live demonstrations for vulnerabilities, knowledge quizzes to support learning, expanded testing capabilities, and improvements to navigation. These additions helped make the platform more complete and easier to use, both for learning and for hands-on interaction.

The only limitations in functionality come from factors outside the platform itself. The system relies on external language models and APIs, whose behaviour can change over time as providers update their systems. This can affect outputs or responses in ways that are not fully predictable. However, this dependency was known from the start and is a normal part of working with modern AI systems, rather than a flaw in the implementation.

Rating: great. We delivered all planned features and added meaningful improvements, with the only variability coming from external systems beyond our control.

Stability

The backend is supported by a strong automated test suite, with over 300 tests and more than 96% test coverage. All tests pass in a standard setup, giving us high confidence that the system behaves as expected.

On the frontend, we tested all main user flows carefully. We used the platform ourselves, shared it with our industry partner, Avanade, and also gave it to external users to try. Across all of these, we did not encounter any issues or crashes in normal use.

The main improvement we would add is clearer in-app feedback when something fails due to external API errors. At the moment, these cases are handled and messages are shown, but they could be more explicit and helpful.

Rating: very good. The platform is stable in normal use, with only minor improvements needed in user-facing error messages.

Efficiency

We designed the platform to keep model usage as low as possible while still delivering good results. We simplified where model calls are needed and avoided unnecessary requests, so the system stays fast and responsive.

For features such as the prompt enhancer and the chatbot, we use lower-cost models by default that are still capable of handling the task well. This allows users to interact with these features without high usage costs. At the same time, users can choose to switch to more advanced models if they want more complex outputs.

This approach keeps both latency and cost under control in normal use.

One improvement for the future would be to support parallel execution for more complex testing scenarios. Running multiple tests at the same time would reduce the total time needed to complete a full testing audit and return results to the user more quickly.

Rating: very good. The platform is efficient for its intended use, with clear opportunities to improve performance further for advanced scenarios.

Compatibility

The web app runs in modern browsers and works well on desktop for daily use.

Users can connect different models, either from common cloud providers or from their own endpoints, such as self-hosted models or small services. The platform is not tied to a single provider: you choose and configure the models that fit your setup, so one application can work with the providers you already use or with models you run yourself.

The design mainly focuses on desktop use. While it can run on mobile, the layout has not been fully optimised for smaller screens.

Rating: good. Desktop experience is strong and model setup is flexible, but mobile use could be improved.

Maintainability

We reuse code where possible, so shared logic is kept in one place and does not need to be fixed in multiple files.

Adding new vulnerabilities or content currently requires changes in the codebase. This works well for technical users. Less technical contributors can still help, but they would need access to the repository and some guidance.

There is no CMS, as all structured content is stored in the repository. For a technical team, this works well because it provides clear version history and a single source of truth. Avanade was comfortable with this approach. If the product later needs frequent updates from people who do not edit code, moving to a CMS would be a natural next step; we already flag that in our future work in case that becomes a priority in the future.

We also keep integrations with external AI providers in a single place, so updates are easier to manage if APIs change.

Rating: good. The project is clear and easy for developers to extend, though it is less accessible for non-technical contributors (not expected to make changes to the platform at this stage).

Project Management

We managed the project in a structured and consistent way throughout. We stayed on track with all deadlines and shared regular progress updates with our UCL supervisors and our industry partner, Avanade. We also scheduled weekly meetings to share progress and get feedback, while keeping them short and focused to respect our partner’s time.

Internally, we used planning documents to define responsibilities and track tasks. We also shared frequent updates through our group chat, which helped keep everyone aligned and aware of progress.

We successfully delivered everything we set out to build, along with additional features, within the timeline.

Work was divided so that each person focused on specific parts of the platform. While this worked well for efficiency, it meant we had less direct involvement across other areas of the project. That said, we maintained strong collaboration through code reviews, requiring at least two approvals before merging any changes.

Rating: very good. We stayed organised, communicated well, and delivered beyond the original scope, with only minor room to improve cross-team involvement.

Future work

When we finished the build, we listed possible next steps we might return to later.

You can already try different inputs and watch how the model’s behaviour changes. Further work could make the tool more useful for testing mitigations and for routine security work. It should also stay usable in teaching—for example IGCSE or A-level classes—or for people who are new to security.

Many people now ship software with help from AI coding tools but without much security training. Learning how prompt injection works early can reduce the chance of serious mistakes on live systems. That includes non-technical founders who are building their own products.

The list below is split into themes. It is a set of ideas for later, not a promise of what will be built next.

Learning, testing & demos

Mobile-friendly version

A simpler layout or a cut-down mobile view so browsing and quick tests feel natural on a phone. The current product targets desktop first.

Compare two mitigation stacks

Run the same prompt with two different mitigation setups. Show the risk scores and full model replies next to each other so you can see what changed.

Guided learning paths

Ready-made paths through the content, such as a short intro or a longer deep dive, so new users do not have to face the full catalogue at once.

Prompt autopsy

After a test, show which parts of the prompt probably pushed the score up or down, and suggest what to delete or change. People see a reason, not only a single score.

Model matchup

Send the same test to two models or providers. Compare refusals, leaks, and tone—useful for seeing how the same prompt behaves on different systems.

Read-only demo mode

Use fixed scenarios and scripted answers so a demo runs without real API keys. Useful in classrooms, meetups, or networks where keys are not allowed.

Exportable lesson packs

Pack vulnerabilities, mitigations, and suggested tests into one download (for example a ZIP file or a printable sheet). Someone running a session can reuse the pack instead of rebuilding materials each time.

Community, content & operations

Contributor checklist

A short checklist for new pages: required fields, an example prompt, a risk label. That keeps new entries in the same shape as older ones.

Community knowledge base

Allow community guides and tips next to the official pages, with review before publication and a line on where the information came from.

Content management system (CMS)

Let staff edit vulnerabilities, mitigations, and help pages in a normal content screen instead of changing code.

Real incident footnotes

Under each vulnerability, add a short real-world or public example so the pattern is easy to picture without a long case write-up.

Federated knowledge

Let organisations choose to add anonymised mitigations and lesson notes to one shared search index. Joining would be optional for each organisation.

Community model directory

A place where people can share prompt-injection tests they built or ran: the prompts, mitigations, and what the model did. Others could comment on how safe or risky that setup felt—for example whether the model refused cleanly, leaked data, or was easy to nudge off policy. Over time that builds practical guidance on which models to use for serious testing, alongside vendor specs and response-time notes.
