UI Design

This page documents the design principles, interface evolution, and final UI of the Prompt Injection Protection platform. It combines the HCI design process with the implemented interface evidenced by the current frontend.

Design Principles

The platform was designed to satisfy the project requirements of being a clear, intuitive, web-based system usable by both technical and semi-technical users. Because the product combines education, explanation, experimentation, and reporting, the interface had to balance simplicity with depth.

Intuitiveness & Learnability

A new user should understand "what this is" and "what to do next" quickly, with minimal instructions. We achieved this through clear information architecture with distinct routes (Home, Vulnerability detail, Mitigations, Testing, Prompt Enhancer, Security Assistant), a catalogue-driven navigation that operates as a predictable "list → detail" learning pattern, and a natural reading path supported by Previous/Next vulnerability navigation.

This principle is crucial because the platform must support both users who already understand AI security and users encountering prompt injection concepts for the first time.

Consistency & Visual Coherence

Consistency was essential because the platform contains multiple views with different levels of technical detail. We kept a shared visual language through glass-style panels, rounded card shapes, gradient heading treatment, and repeated page structure. We chose shadcn/ui (built on Radix) with Tailwind CSS to ensure consistent primitives—dialogs, tabs, sheets, and other components behave predictably across the UI.

Styling rules enforce consistency: a cn() helper, semantic colour tokens, preference for utilities over custom CSS, and mandatory dark-mode parity. This prevents one-off styling and makes the interface feel cohesive even as it scales.

Usability & Efficiency

Users should be able to iterate quickly and complete common tasks without repeated work. The Home page features real-time search and impact-level filter pills (All/High/Medium/Low) for rapid scanning of content. The Testing view supports reusable testing workflows through Test Actions (duplication, modification, removal) and Test Configurations (save and apply reusable setups).

We also included an auto-play demo mode for the terminal, reducing interaction cost for passive learning and presentation scenarios.

Accessibility & Inclusive Design

Users should be able to use the UI across themes, devices, and interaction modes (mouse/keyboard). Dark mode is a non-negotiable requirement, implemented system-wide through semantic colour tokens and theme variables in frontend/src/styles/theme.css, rather than one-off styling decisions.

Responsive design is built in from the start: a mobile header collapses to a hamburger menu and sheet under 768px, supporting usability across devices. This approach makes accessibility practical and maintainable, not an afterthought.

Design rationale: the interface was intentionally designed to feel educational, practical, and exploratory at the same time. Users should be able to move from “what is this?” to “how does it work?” to “how do I test it?” in a natural way, without leaving the system.

Initial Sketches

The initial sketches explored the core information architecture of the platform before any implementation began. At this stage, the emphasis was on layout, navigation, and how to present vulnerabilities, mitigations, and testing workflows in a way that would be understandable to both technical and semi-technical users.

The main goal of the initial sketches was to validate the broad structure of the application: a clear landing area, deep-dive content views, and separate spaces for practical testing and assistance.

Refined Sketches

The refined sketches capture a clear design pivot from broad early concepts to focused user journeys. We prioritised helping users understand prompt injection faster on the Home page, using richer explanatory content and visual resources such as statistics.

We also restructured the Vulnerabilities area around an intuitive with-vs-without-mitigation behaviour comparison, and moved practical experimentation into a dedicated Testing workspace. This separation made the product goals clearer: learn concepts first, compare outcomes clearly, and run organised tests in a purpose-built environment.

Initial Prototype

In the initial prototype, we focused on making the panel structure easier to understand and simpler to navigate. Each major section used distinct color cues to improve intuitiveness and accessibility, helping users recognize context and purpose at a glance.

The goal of this version was to establish a clearer information hierarchy across views while keeping interactions approachable for both technical and semi-technical users.

Final UI

The final implemented UI is a React single-page application with dedicated routes for the major platform workflows. Rather than forcing every feature into one screen, the final design uses a task-oriented structure: the Home page introduces the problem and vulnerability catalogue, the Vulnerability view provides a deep technical breakdown, the Mitigations page focuses on defensive strategies and code, the Testing page supports practical experimentation, the Prompt Enhancer hardens prompts, and the Security Knowledge Assistant supports open-ended Q&A.

Final design process

The final UI reflects a progression from broad educational structure to task-specific views. The implemented frontend shows that the design prioritised modularity, with each route solving a distinct user need while still sharing a common visual system. This is important because the platform needs to support both learning and action: a user may begin by reading about a vulnerability, then move to mitigations, then run a test, then export a report, then ask the assistant for more help. We also introduced a dedicated dark mode to improve accessibility and visual comfort, making the interface more adaptable to different user preferences and usage environments.

Feedback reflected in the final structure

During the mid-stages of development, we received feedback from our TA that the interface felt too static and old-fashioned. In response, we pivoted to a more modern and dynamic visual direction based on widget-style components rather than heavy panel blocks, and refined the color system and overall styling to be clearer and less visually intrusive. This change improved both readability and perceived responsiveness while keeping the workflow structure intact.

Personas & User Needs

Our platform was intentionally designed to serve both technical and semi-technical users without forcing either group to navigate unnecessary complexity. We achieved this by structuring distinct workflows: learning, investigation, testing, and knowledge support. Each with layered explanations that allow users at any experience level to engage at their comfort level. This modular approach lets users distinguish between essential information and advanced features, empowering them to focus on what matters to their role and goals.

Accessibility

Accessibility was treated as a core requirement from the start. The interface supports keyboard navigation with visible focus indicators, responsive design across mobile and desktop, system-wide dark mode, semantic colour coding (red for vulnerabilities, green for mitigations, orange for actions), and a clear information hierarchy that remains scannable across all page sizes.

Keyboard Navigation

All interactive elements accessible via keyboard with visible focus indicators and logical tab order.

Responsive Design

Fully responsive layouts tested across mobile, tablet, and desktop viewport sizes.

Colour & Contrast

Colour is never the sole indicator of meaning; all text meets WCAG AA contrast ratios.

Screen Reader Support

Semantic HTML, ARIA labels, and meaningful alt text for full screen reader compatibility.
