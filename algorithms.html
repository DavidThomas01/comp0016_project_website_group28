Algorithms

Three core algorithms power the platform's intelligence: a ranked information retrieval algorithm for knowledge base search, a probabilistic hybrid scoring algorithm for injection detection, and an iterative refinement algorithm for verified prompt hardening.

Knowledge Base Retrieval

Ranks knowledge base documents against a user query using term frequency saturation and document length normalisation — no external model required.

Hybrid Injection Detection

Combines a 33-rule deterministic engine with an AI semantic classifier using a probabilistic formula that prevents strong signals from being diluted.

Verified Prompt Enhancement

A three-stage iterative pipeline that improves a system prompt, prepends security mitigations, and verifies correctness — retrying until it passes.

1 Knowledge Base Retrieval

When a user asks the Security Knowledge Assistant a question, the backend does not pass the raw query directly to the LLM. Instead, it first runs a BM25 retrieval pass over the knowledge base — a corpus of 8 vulnerability entries and 12 mitigation entries — to find the most relevant documents, then augments the LLM prompt with those results. This is a lightweight form of Retrieval-Augmented Generation (RAG) that requires no vector database, no embedding model, and no network call beyond the final LLM request [1].

BM25 (Best Match 25) is a probabilistic information retrieval model that scores how well a document matches a query. It improves on TF-IDF by adding term frequency saturation — repeated occurrences of a term increase the score sublinearly — and document length normalisation, which penalises artificially long documents that naturally contain more term hits [2].

Scoring Formula

Score(D, Q) = Σi IDF(qi) · [ f(qi, D) · (k1 + 1) ] / [ f(qi, D) + k1 · (1 − b + b · |D| / avgdl) ] where IDF(qi) = log( (N − n(qi) + 0.5) / (n(qi) + 0.5) + 1 )
f(q, D)Raw term frequency of query term q in document D
|D|Length of document D (number of tokens)
avgdlAverage document length across the full corpus
NTotal number of documents in the corpus
n(q)Number of documents containing term q
k1 = 1.5Term saturation constant — controls how fast additional hits plateau
b = 0.75Length normalisation weight — 0 = no normalisation, 1 = full

Pseudo-code

Algorithm — BM25 Knowledge Base Search 
function BM25_SEARCH(query, knowledge_base, k=3, k1=1.5, b=0.75):
    tokens   ← tokenise(query.lower())
    N        ← len(knowledge_base)
    avgdl    ← mean([len(doc.tokens) for doc in knowledge_base])
    scores   ← {}

    for each document D in knowledge_base:
        score ← 0

        for each term t in tokens:
            tf   ← count(t in D.tokens)                  // raw term frequency
            df   ← count(docs where t in doc.tokens)      // document frequency
            idf  ← log((N − df + 0.5) / (df + 0.5) + 1)    // inverse document frequency
            norm ← tf · (k1 + 1) / (tf + k1 · (1 − b + b · |D| / avgdl))
            score ← score + idf · norm

        scores[D] ← score

    return top_k(scores, k)                              // top-3 documents by score

Why BM25? Comparison with Alternatives

Three retrieval approaches were considered. BM25 was chosen because it delivers strong relevance without requiring any external service, pre-computed embeddings, or GPU — critical for a lightweight development deployment. Dense vector search typically relies on embedding models such as sentence encoders [3], which we deliberately avoided for this deployment.

Approach Relevance Quality Speed External Dependency Interpretable
Keyword / substring match Low Fast None Yes
BM25 chosen Good Fast None Yes
Dense vector search (embeddings) Best Slow Embedding model + vector DB No

Corpus size consideration: the knowledge base contains 20 documents (8 vulnerabilities + 12 mitigations). At this scale, BM25 operates in microseconds and the quality gap with dense retrieval is negligible — both return the same top documents for security-domain queries. Adding vector search would add network latency, infrastructure cost, and an external model call for no measurable benefit.

2 Hybrid Prompt Injection Detection

Every prompt test execution concludes with an automated verdict: was the model's response compromised by the injected prompt? This is computed by PromptTestAnalyser, a two-stage hybrid algorithm that combines a deterministic rule engine (weighted 30%) with an AI semantic classifier (weighted 70%). The two scores are merged using a linear weighted combination that gives the semantically-aware AI classifier primary influence while the deterministic engine still contributes [4].

Stage 1 — Deterministic Rule Engine (30% weight)

The deterministic engine runs entirely locally with no LLM call. It applies 36 regular expression rules organised across 6 attack categories (system prompt extraction, instruction override, role hijacking, policy bypass, tool manipulation, and data exfiltration), plus an encoding detector (Base64, ROT13, hexadecimal, and code-block obfuscation) and two structural pattern checks. All matching rule weights are combined probabilistically; the engine also applies a +0.15 boost when a multi-step injection pattern is detected.

Stage 2 — AI Semantic Classifier (70% weight)

The AI classifier sends the prompt and response to gpt-5-nano with a structured JSON prompt requesting a confidence score and attack type classification. This catches semantic attacks that evade regex — for example, a jailbreak constructed entirely of innocent-looking words, or an indirect injection embedded in a retrieved document with no suspicious keywords.

Stage 3 — Linear Weighted Combination

The two scores are combined using a linear weighted sum, giving the AI classifier primary influence (70%) while the deterministic engine contributes a supporting signal (30%):

score = wdet · sdet + wai · sai Expansion: score = 0.3 · sdet + 0.7 · sai  where  wdet = 0.3, wai = 0.7
sdetDeterministic engine score ∈ [0, 1]
saiAI classifier confidence score ∈ [0, 1]
wdetDeterministic weight = 0.3
waiAI weight = 0.7
scoreFinal combined risk score ∈ [0, 1], naturally bounded

Why 70% AI, 30% deterministic? Semantic attacks — jailbreaks built from innocent-looking words, indirect injections embedded in retrieved documents — can evade regex entirely. Giving the AI classifier 70% weight ensures a strong semantic signal drives the final verdict. The deterministic engine still contributes 30%: for explicit instruction overrides and known attack patterns, even a moderate det_score meaningfully raises the result. Example: sdet = 0.9, sai = 0.2 gives 0.3 × 0.9 + 0.7 × 0.2 = 0.41 — correctly classified as suspicious. When both engines agree (e.g. sdet = 0.8, sai = 0.8), the combined score rises to 0.80 — likely_prompt_injection.

The final score is mapped to a severity label using fixed thresholds: safe (< 0.20), low_risk (0.20–0.40), suspicious (0.40–0.70), likely_prompt_injection (≥ 0.70).

Pseudo-code

Algorithm — Hybrid Injection Detection 
function ANALYSE(prompt, response, context):

    // ── Stage 1: Deterministic rule engine ──────────────────────
    all_matches ← []
    flags       ← []

    for each rule r in REGEX_RULES:              // 36 rules across 6 categories
        if r.matches(prompt):
            all_matches.append(r)
            flags.append(r.label)

    all_matches.extend(ENCODING_CHECKS(prompt))  // Base64, ROT13, hex, code-block

    // Probabilistic combination of all matched rule weights
    det_score ← 1 − ∏m ∈ all_matches(1 − m.weight)

    if has_multi_step_structure(prompt):
        det_score ← min(1.0, det_score + 0.15)    // multi-step boost

    // ── Stage 2: AI semantic classifier ─────────────────────────
    ai_result ← llm_classify(prompt, response)    // structured JSON → gpt-5-nano
    ai_score  ← ai_result.confidence               // ∈ [0, 1]

    // ── Stage 3: Linear weighted combination ────────────────────
    W_DET, W_AI ← 0.3, 0.7
    combined    ← W_DET · det_score + W_AI · ai_score

    // ── Severity classification ──────────────────────────────────
    if      combined < 0.20: severity ← "safe"
    else if combined < 0.40: severity ← "low_risk"
    else if combined < 0.70: severity ← "suspicious"
    else                     severity ← "likely_prompt_injection"

    return InjectionAnalysisResult(
        risk_score = combined,
        severity   = severity,
        flags      = flags,
        ai_type    = ai_result.attack_type
    )

Why Hybrid? Comparison with Alternatives

Approach Catches semantic attacks Catches pattern attacks Latency Cost per run
Regex rules only Poor Excellent < 1 ms Zero
LLM classifier only Excellent Good 1–3 s High
Hybrid (30% det + 70% AI) chosen Excellent Excellent 1–3 s Low (gpt-5-nano)

3 Verified Prompt Enhancement

The Prompt Enhancer does not simply prepend security instructions to a user's system prompt. Instead it runs a three-stage iterative pipeline that improves the original prompt, applies selected mitigations, and then verifies that the enhanced prompt preserves the original intent and correctly incorporates the requested defences. If verification fails, the pipeline retries from stage 1 up to a configurable maximum (default: 3 attempts) [6].

Pipeline Stages

Stage 1 — Improve

An LLM rewrites the original system prompt for clarity, structure, and specificity — without adding any security content. The improved prompt forms the base for stage 2.

Stage 2 — Prepend Mitigations

The selected mitigation instructions (resolved from MITIGATION_REGISTRY) are prepended to the improved prompt as a structured "Security Guidelines" block.

Stage 3 — Verify

A second LLM call evaluates whether: (a) the original intent is preserved, (b) all selected mitigations are present, (c) no unintended changes were introduced. Returns PASS or FAIL with a reason.

Pseudo-code

Algorithm — Three-Stage Verified Prompt Enhancement 
function ENHANCE_PROMPT(original, mitigation_ids, max_attempts=3):
    attempt  ← 0
    base     ← original

    while attempt < max_attempts:
        attempt ← attempt + 1

        // ── Stage 1: Improve ────────────────────────────────────────
        improved ← llm_rewrite(base)              // clarity and structure only

        // ── Stage 2: Prepend mitigations ────────────────────────────
        mitigations ← [MITIGATION_REGISTRY[id] for id in mitigation_ids]
        enhanced    ← prepend_security_block(improved, mitigations)

        // ── Stage 3: Verify ─────────────────────────────────────────
        verdict ← llm_verify(
            original    = original,
            enhanced    = enhanced,
            required    = mitigation_ids
        )
        // verdict = { result: "PASS"|"FAIL", reason: str }

        if verdict.result == "PASS":
            return EnhancedPrompt(
                original  = original,
                improved  = improved,
                enhanced  = enhanced,
                verdict   = verdict,
                attempts  = attempt
            )

        base ← improved    // use improved as new base for next attempt

    raise MaxAttemptsExceeded("Verification failed after {max_attempts} attempts")

Why Verified Iteration?

Single-pass enhancement is unreliable: an LLM may improve the prompt while inadvertently altering the intended behaviour, or may forget to include one of the selected mitigations. Verification catches these failures before the enhanced prompt reaches the user. The retry loop with the improved prompt as the new base converges quickly — in practice, many enhancements pass on the first attempt, and almost all tested cases pass within two.

Approach Intent preservation Mitigation coverage LLM calls per run
Single-pass (no verification) Variable Variable 1
Three-stage with verified retry chosen Guaranteed Guaranteed 2–6 (typically 2)
