1. Project Summary & Features

The Prompt Injection Protection platform is a full-stack educational security tool built for Avanade. Its purpose is to help developers and security professionals understand, reproduce, test, and defend against prompt injection vulnerabilities in Large Language Models. Rather than describing attacks abstractly, the platform lets users run real prompts against real models and observe exactly how mitigations change the outcome.

Vulnerability Catalogue

Eight prompt injection attack types explained in depth, each with an interactive demo terminal, example attack strings, and a knowledge-check quiz.

Mitigation Library

Twelve security defences organised across four pipeline layers (PROMPT, PRE_INPUT, POST_OUTPUT, MONITORING), each with technical descriptions and usage context.

Testing Workbench

Configure a model, write a system prompt, select mitigations, submit a prompt, and receive a risk-scored analysis — all in one interface.

Red-Teaming Framework

Integration with Garak — an industry-standard LLM red-teaming framework — enabling automated multi-probe adversarial testing against any configured model.

Prompt Enhancer

Input any system prompt and selected mitigations; the platform runs a three-stage AI pipeline to harden and verify it, returning a side-by-side comparison.

Security Knowledge Assistant

A streaming chat assistant augmented with BM25 retrieval over the vulnerability and mitigation knowledge base — ask anything about prompt injection.

API key setup: all LLM calls route through Microsoft Azure AI Foundry. Users must supply their own per-model credentials via a .env file in the backend directory. Each model requires a FOUNDRY_*_ENDPOINT and a FOUNDRY_*_KEY environment variable. No keys are bundled with the repository. A .env.example file documents the full set of required variables.

4. Frontend & User Interaction

The frontend is a single-page application with six routes and a floating chat widget available on all pages. Every interaction that requires data or model inference is delegated to the backend via typed API calls. The frontend holds no credentials and makes no direct LLM requests.

Vulnerability Catalogue HomePage.tsx + VulnerabilityPage.tsx
What it is

A browsable catalogue of all prompt injection vulnerability types, with quick discovery on the main page and full technical walkthroughs on each vulnerability page.

How it was implemented

Vulnerability metadata is stored in static TypeScript data (data/vulnerabilities.ts) and rendered as cards on HomePage.tsx. Search/filter logic uses a controlled input and client-side Array.filter(). Route-based detail rendering in VulnerabilityPage.tsx resolves entries by slug via React Router.

Additional details & features
  • Vulnerability detail: technical explanation, attack anatomy, and defence rationale for each type.
  • Demo terminal: auto-playing simulated attack/defence flow with typewriter-style output.
  • Quiz: multiple-choice knowledge-check questions with immediate feedback and score state.
  • Navigation: previous/next vulnerability navigation for guided learning progression.
Testing Workbench TestingPage.tsx
What it is

The main interactive page for configuring a model + mitigations, running prompt tests, and receiving risk analysis outputs in one place.

How it was implemented

The page sends CreateTestRequest payloads to /api/v1/tests and executes runs via /api/v1/tests/:id/run. Model configuration supports both PLATFORM and EXTERNAL types, and risk is visualised as a 0–1 gradient gauge in the UI.

Additional details & features
  • Model selection: pick from platform registry models or define external endpoint models.
  • Mitigation toggles: enable/disable defences per run to compare security impact.
  • Runner choice: execute tests through PromptRunner or Garak depending on testing mode.
  • Result panel: shows score, severity, flagged reasons, and mitigation-level output analysis.

What it is

A hardening workflow that improves a system prompt, injects selected mitigations, and verifies the enhanced prompt before returning it to the user.

A single call to /api/v1/enhancements triggers the full backend enhancement pipeline. The frontend renders original, improved, and enhanced prompts side-by-side and displays verification outcome using status-aware badges.

  • Prompt input: accepts arbitrary system prompts from users.
  • Mitigation selection: users choose which mitigations to embed into the enhanced result.
  • Three-stage output: displays original, improved, and final enhanced prompt text.
  • Verification metadata: shows PASS/FAIL verdict, reason, and attempt count.

What it is

A streaming assistant that answers prompt-injection and mitigation questions, available as both a dedicated page and a floating widget across the app.

The frontend opens an SSE stream to POST /api/v1/chat/stream and appends token deltas in real time. Chat history is persisted with localStorage, and the shared ChatWidget is mounted in the root layout for cross-route access.

5. Backend Implementation

The backend is a Python 3.12 FastAPI application structured into four strict layers — API, Application, Domain, and Infrastructure. Each major capability is described below with the relevant source code and the reasoning behind key implementation decisions.

Model Providers Infrastructure

All LLM access flows through a single abstract base class, ModelProvider, defined in the domain layer. Any class that implements generate() becomes a valid provider — the rest of the system never knows which concrete class it is talking to.

Base class — domain/providers/base_provider.py
class ModelProvider(ABC):

    @abstractmethod
    async def generate(self, request: ModelRequest) -> ModelResponse:
        """Execute a model call and return a unified response.
        Must not raise provider-specific exceptions."""
        pass

Each entry stores the provider type (which tells ProviderRouter which class to instantiate), the endpoint env var name, the API key env var name, and the model's wire-format name. No credentials are hardcoded — all values are looked up from environment variables at request time.

Three concrete implementations live in the infrastructure layer, registered in _PROVIDER_REGISTRY and instantiated on demand by ProviderRouter:

OpenAI-Compatible Provider — GPT & open-weight models via Azure AI Foundry

OpenAICompatibleProvider handles GPT-5.2, GPT-5.1, GPT-5-nano, o4-nano, Llama 3.3-70B, Phi-4, DeepSeek V3.1, and Mistral Small 2503 — all hosted on Azure AI Foundry and accessed through the OpenAI-format chat completions API. Endpoint URL and API key are looked up from environment variables whose names are stored in MODEL_REGISTRY. Azure requires the api-key header rather than the standard Authorization: Bearer. 

async def generate(self, request: ModelRequest) -> ModelResponse:
    config   = MODEL_REGISTRY.get(request.model)
    endpoint = os.getenv(config.endpoint)   # env var name from registry
    api_key  = os.getenv(config.api_key)

    messages = self._build_messages(request)  # prepend system prompt if present
    payload  = {
        "model":    config.model_name,
        "messages": [{"role": m.role, "content": m.content} for m in messages],
    }
    headers = {"Content-Type": "application/json", "api-key": api_key}

    async with httpx.AsyncClient(timeout=self._timeout) as client:
        response = await client.post(endpoint, json=payload, headers=headers)

    data = response.json()
    return ModelResponse(text=self._extract_text(data), raw=data)

_extract_text handles both choices[0].message.content (standard completions) and output_text (Azure Responses API format) so the same provider works with all Foundry endpoints without format negotiation.

Anthropic Provider — Claude models via Anthropic API

AnthropicProvider handles Claude Sonnet 4.5, Haiku 4.5, and Opus 4.1. The Anthropic Messages API differs from OpenAI in two key ways that require separate handling: the system prompt is a top-level "system" field rather than a role: "system" message, and every request requires an explicit max_tokens field. Auth uses x-api-key plus a mandatory anthropic-version header. 

def _build_messages(self, request) -> Tuple[str | None, List[Message]]:
    # Anthropic: system prompt is a top-level field, not a message role
    system_prompt = request.system_prompt
    messages = [m for m in request.messages if m.role != "system"]
    return system_prompt, messages

payload = {
    "model":      config.model_name,
    "messages":   [{"role": m.role, "content": m.content} for m in messages],
    "max_tokens": request.metadata.get("max_tokens", 1024),  # required field
}
if system_prompt:
    payload["system"] = system_prompt   # top-level, not inside messages[]

headers = {
    "x-api-key":         api_key,
    "anthropic-version": "2023-06-01",
}
External HTTP Provider — arbitrary user-supplied endpoints

ExternalHttpProvider allows testing against any HTTP endpoint — internal company LLMs, locally-hosted models, or custom deployments. The user supplies an endpoint URL, optional custom headers, the payload field name for the message (message_field), and a dot-path to extract the response text (response_text_path). An optional json_schema is validated against the outgoing payload before the request is sent. 

async def generate(self, request: ExternalModelRequest) -> ModelResponse:
    payload = self._build_payload(request)      # inject message into user-supplied template
    self._validate_payload(payload, request.json_schema)  # optional JSON Schema check

    async with httpx.AsyncClient(timeout=self._timeout) as client:
        response = await client.post(request.endpoint, json=payload,
                                        headers=self._build_headers(request))

    text = self._extract_response_text(response.json(), request.response_text_path)
    return ModelResponse(text=text, raw=response.json())

Conversation mode "single" sends only the last message as a plain string; mode "conversation" sends the full history array — configurable per endpoint to match whatever schema the target API expects.

Why one ABC: every service that needs a model holds only a ModelProvider reference. Adding a new provider (e.g. Google Gemini) requires one new class and one registry entry — zero changes to ProviderRouter, ChatService, PromptRunner, or any other consumer.
Test Domain Object Domain

Test is the central domain entity. It encodes everything needed to execute a prompt-injection test: which model to target, what environment (system prompt + mitigation slugs) to apply, and which runner to use. All construction goes through a static factory method that enforces domain invariants before the object is accepted.

Entity — domain/tests/test.py
@dataclass
class Test:
    id:          str                          # "test_" + 10 random hex chars
    name:        str
    model:       ModelSpec                  # PLATFORM | EXTERNAL + model_id / endpoint
    environment: Optional[EnvironmentSpec]   # system_prompt + List[mitigation_slug]
    runner:      RunnerSpec                  # PROMPT | FRAMEWORK + conversation history
    created_at:  datetime

    @staticmethod
    def create(name, model, environment, runner) -> "Test":
        Test._validate(model=model, environment=environment, runner=runner)
        return Test(id=new_test_id(), name=name, model=model,
                       environment=environment, runner=runner, created_at=datetime.now())

    def update(self, *, name=None, model=None, environment=None, runner=None) -> "Test":
        # Returns a new Test — the original instance is never mutated
        Test._validate(model=new_model, environment=new_env, runner=new_runner)
        return Test(id=self.id, ...)

Domain rules enforced at create() and update() time:

  • A FRAMEWORK runner (Garak) requires a PLATFORM model — external black-box endpoints cannot be red-teamed via Garak.
  • An EXTERNAL model cannot carry an EnvironmentSpec — system prompt and mitigation pipeline are meaningless for a user-supplied HTTP endpoint.
  • A PLATFORM model with a PROMPT runner requires an EnvironmentSpec — the system prompt and mitigation list are mandatory.
Why a factory + immutable update: create() is the only entry point for a valid Test. update() produces a new instance rather than mutating in place, so no partially-invalid intermediate state can exist during an update operation. Repository writes only ever receive fully-validated objects.
Test Runner Infrastructure

TestRunner is an ABC with a single method. PromptRunner handles interactive prompt tests; GarakRunner handles automated red-teaming. Both implement the same interface, so RunnerRouter can resolve and call them interchangeably at runtime.

Abstract base — domain/tests/test_runner.py
class TestRunner(ABC):
    @abstractmethod
    async def run(self, test: Test, prompt: Optional[Message]) -> TestResult:
        pass
PromptRunner — infra/tests/runners/prompt_runner.py
async def run(self, test: Test, prompt: Optional[Message]) -> TestResult:
    started_at = datetime.now()

    if test.model.type == ModelType.PLATFORM:
        # Full mitigation pipeline: PRE_INPUT → LLM call → POST_OUTPUT → MONITORING
        model_response, mitigation_analysis = \
            await self._run_test_on_platform_model(test, prompt)

    elif test.model.type == ModelType.EXTERNAL:
        # No mitigation pipeline — direct call to user-supplied endpoint
        model_response = await self._run_test_on_external_model(test, prompt)

    analysis = await self.analyser.analyse(
        model_response, prompt, test.runner.context, mitigation_analysis
    )
    return TestResult(
        output=model_response.text, analysis=analysis,
        started_at=started_at, finished_at=datetime.now()
    )

For platform models, _run_test_on_platform_model applies mitigations in strict layer order. PRE_INPUT mitigations (input validation, blocklist filtering, pattern matching) run first on the raw prompt. PROMPT-layer mitigations are compiled into the system prompt by PromptCompiler — each contributes its prompt_message as a security instruction block. After the model call, POST_OUTPUT mitigations sanitise the response, and MONITORING mitigations return per-mitigation anomaly scores that feed into the final analysis.

GarakRunner — automated red-teaming via managed subprocess

GarakRunner writes a temporary YAML configuration file encoding the selected model, probe type, and detector, then launches garak as a managed subprocess via asyncio.create_subprocess_exec. stdout is captured and parsed as JSONL hit reports. A DEFCON-aggregated score is computed from the reports and returned as a TestResult. Garak's generated HTML report is served statically by FastAPI and its URL is included in the response so the user can open it directly in the browser.

Garak runs as a subprocess rather than a Python import because its large dependency tree conflicts with FastAPI's async event loop. Subprocess isolation also means Garak version upgrades require no changes to the backend's own dependency graph.

Why one ABC for two very different runners: TestService calls runner.run(test, prompt) and receives a TestResult — it has no knowledge of whether Garak or PromptRunner handled the request. Adding a third runner (e.g. PyRIT) requires one new class and one registry entry with zero service-layer changes.
Mitigation System — Layered Defence Pipeline Domain + Infrastructure

Each mitigation is a class that implements a single-method ABC: apply(context) → context. The context carries the current message, conversation history, arbitrary metadata, and an optional MitigationAnalysis result. Mitigations are organised into four layers that determine when in the request lifecycle they run.

Base types — domain/mitigations/mitigations_base.py
class MitigationLayer(Enum):
    PROMPT      = "prompt"       # injected into the system prompt before the LLM call
    PRE_INPUT   = "pre_input"    # runs on the user's message before it reaches the model
    POST_OUTPUT = "post_output"  # runs on the model's response before it reaches the user
    MONITORING  = "monitoring"   # observes the full interaction; produces an anomaly score

class Mitigation(ABC):
    layer: MitigationLayer

    @abstractmethod
    def apply(self, context: MitigationContext) -> MitigationContext:
        pass

@dataclass(frozen=True)
class MitigationConfig:
    id:             str
    name:           str
    layer:          MitigationLayer
    prompt_message: str | None    # PROMPT-layer text; None for runtime mitigations
    implementation: Mitigation | None  # None = prompt-only, no runtime class needed

All 12 mitigations are declared in MITIGATION_REGISTRY — a flat dictionary mapping slug → MitigationConfig. Seven are PROMPT-layer (instruction-text only, no runtime class), three are PRE_INPUT (active filters), one is POST_OUTPUT, and one is MONITORING.

Registry excerpt — infra/config/mitigations.py

PROMPT layer — how instruction text gets into the system prompt

PROMPT-layer mitigations have no apply() method — they are pure text. PromptCompiler iterates the selected mitigation slugs, reads the prompt_message from each MitigationConfig, and appends it to the system prompt under a "For security, you must follow these instructions:" header before the model call. 

class PromptCompiler:
    @staticmethod
    def compile(mitigations: List[str], system_prompt: Optional[str] = None) -> str:
        system_prompt += "For security, you must follow these instructions:"
        for slug in mitigations:
            config = MITIGATION_REGISTRY.get(slug)
            if config.layer == MitigationLayer.PROMPT:
                system_prompt += "\n" + config.prompt_message
        return system_prompt
PRE_INPUT layer — PatternMatchingMitigation example

The most illustrative PRE_INPUT mitigation is PatternMatchingMitigation. It runs 8 compiled regex patterns against the incoming prompt (covering phrases like "ignore previous instructions", "reveal your system prompt", "developer mode", etc.) plus a fuzzy matching layer to catch obfuscated variants. Matches are redacted in-place and the mitigation writes an analysis score back onto the context. 

def apply(self, context: MitigationContext) -> MitigationContext:
    sanitized = original_text
    hits = []

    for pattern in self._PATTERNS:   # 8 compiled regex rules
        if pattern.search(sanitized):
            hits.append(pattern.pattern)
            sanitized = pattern.sub("[PROMPT_INJECTION_PATTERN_REDACTED]", sanitized)

    # Fuzzy layer — catches typoglycemia / punctuation obfuscation
    fuzzy_hits = [t for t in self._FUZZY_TARGETS if self._contains_fuzzy(normalized, t)]

    context.analysis = MitigationAnalysis(
        flagged = bool(hits),
        score   = min(1.0, 0.25 * len(hits)),
        reason  = "Prompt-injection pattern detected and sanitized." if hits else "Clean.",
    )
    return context
POST_OUTPUT layer — OutputSanitizationMitigation

OutputSanitizationMitigation runs on the model's raw response before it is returned to the user. It applies 7 targeted regex rules to redact sensitive content that a successfully-injected model might leak: API keys, AWS credentials, generic secrets, system-prompt disclosure phrases, internal instruction leakage, and injectable <script> / javascript: URIs. Matches are replaced in-place with labelled placeholders. 

_REDACTIONS = [
    (re.compile(r"\bsk-[A-Za-z0-9_-]{16,}\b"),                  "[REDACTED_API_KEY]"),
    (re.compile(r"\bAKIA[0-9A-Z]{16}\b"),                         "[REDACTED_AWS_KEY]"),
    (re.compile(r"(?i)\b(?:api[_ -]?key|secret|token|password)\b\s*[:=]\s*\S+"), "[REDACTED_SECRET]"),
    (re.compile(r"(?is)(?:the\s+)?system\s+prompt\s+is\s*:?\s*.+?(?=\n\n|$)"), "[SYSTEM_PROMPT_REDACTED]"),
    (re.compile(r"(?is)<script.*?>.*?</script>"),                    "[SCRIPT_REMOVED]"),
    # ... 2 more rules
]

def apply(self, context) -> MitigationContext:
    sanitized = original_text
    hits = []
    for pattern, replacement in self._REDACTIONS:
        if pattern.search(sanitized):
            hits.append(pattern.pattern)
            sanitized = pattern.sub(replacement, sanitized)   # redact in-place
    self._set_text(message, sanitized)
    context.analysis = MitigationAnalysis(
        flagged = bool(hits),
        score   = min(1.0, 0.25 * len(hits)),
        reason  = "Sensitive output redacted." if hits else "Output passed sanitization.",
    )
    return context
MONITORING layer — AnomalyDetectionMitigation

The MONITORING mitigation runs after the full exchange and scores the entire conversation for behavioural anomalies. It checks for suspicious pattern matches, unusually long interactions, high token repetition, excessive punctuation density, and uppercase overuse — each contributing additively to a 0–1 anomaly score. It never modifies the message; it only writes an analysis. 

score = 0.0
for pattern in self._SUSPICIOUS_PATTERNS:   # 6 heuristic regexes
    if pattern.search(combined): score += 0.15

if len(combined) > 2500:      score += 0.15   # unusually long interaction
if repetition_ratio > 0.22:  score += 0.20   # high token repetition
if punct_density > 0.18:      score += 0.10   # punctuation stuffing
if uppercase_ratio > 0.35:   score += 0.10   # excessive emphasis

context.analysis = MitigationAnalysis(
    flagged = score >= 0.45,
    score   = round(min(1.0, score), 2),
    reason  = ", ".join(reasons),
)

How mitigations are applied in a test run

When PromptRunner executes a platform-model test, it applies all selected mitigations in four ordered passes. The code below shows the actual call sequence inside _run_test_on_platform_model: 

async def _run_test_on_platform_model(self, test, prompt):

    # PRE_INPUT — filter and sanitize the incoming user prompt
    prompt = self._run_pre_input_mitigations(test.environment.mitigations, prompt, context)

    # LLM call — filtered prompt + hardened system prompt
    request  = ModelRequest(model=test.model.model_id,
                             messages=context + [prompt],
                             system_prompt=compiled_system_prompt)
    response = await provider.generate(request)

    # POST_OUTPUT — redact sensitive content from the model's response
    response = self._run_post_output_mitigations(test.environment.mitigations, response, context)

    # MONITORING — score the full exchange for anomalies (read-only)
    mitigation_analysis = self._run_monitoring_mitigations(test.environment.mitigations, response, context)

    return response, mitigation_analysis
Why one ABC for all layers: every runtime mitigation — whether it filters input, sanitises output, or monitors behaviour — implements the same apply(context) → context interface. PromptRunner iterates the slugs for each layer and calls apply() without knowing what the mitigation does. Adding a new mitigation is one new class + one registry entry.
Test Analysis — Hybrid Injection Detection Infrastructure

After every PromptRunner execution, PromptTestAnalyser passes the full conversation transcript through PromptInjectionAnalyzer — a hybrid engine that combines deterministic rule-based detection with AI semantic analysis. The two scores are merged via probabilistic combination into a single risk score (see Algorithms page for the full formula and weighting).

Hybrid analyser — infra/tests/analyzers/analysis/prompt_injection_analyzer.py
class PromptInjectionAnalyzer:
    """30 % deterministic rules + 70 % AI semantic analysis."""

    async def analyse(self, conversation: List[Dict[str, str]]) -> InjectionAnalysisResult:
        # Stage 1: run 33 regex rules across 7 categories
        structural: StructuralAnalysis = analyse_conversation_structure(conversation)
        deterministic_score = compute_deterministic_score(structural)

        # Stage 2: AI semantic classifier (GPT-5-nano)
        ai_result: AIAnalysisResult = await analyse_with_ai(conversation)

        # Stage 3: probabilistic combination → final risk score
        risk_score = compute_risk_score(deterministic_score, ai_result.confidence)

        return InjectionAnalysisResult(
            risk_score        = risk_score,
            severity          = get_severity(risk_score),   # safe/low_risk/suspicious/likely_injection
            deterministic_score = deterministic_score,
            ai_score          = ai_result.confidence,
            flags             = build_flags(structural),
            detected_patterns = build_detected_patterns(structural),
            explanations      = self._build_explanations(structural, ai_result),
        )

PromptTestAnalyser wraps this with an error boundary — if analysis fails for any reason, it returns a zero-score MitigationAnalysis so the test result is still delivered to the frontend. The flagging threshold is a risk score ≥ 0.4 (suspicious severity or above). Explanations carry source prefixes ([ai] vs [category]) so the UI can attribute each finding to its origin.

Why a wrapper layer: PromptInjectionAnalyzer returns the rich internal InjectionAnalysisResult; PromptTestAnalyser maps it to the simpler MitigationAnalysis that PromptRunner expects. Keeping the two types separate means the internal analyser can evolve without touching the runner contract.
Chatbot — BM25 Knowledge Retrieval + SSE Streaming Application

ChatService handles every message in two phases. First it searches the in-memory knowledge base (8 vulnerabilities + 12 mitigations) using BM25 and retrieves the top 3 most relevant documents (see Algorithms page). These are formatted into a markdown context block and prepended to the user's message before the LLM call. The response is then streamed back to the frontend as Server-Sent Events via FastAPI's StreamingResponse.

Knowledge augmentation + SSE stream — app/chat/chat_service.py
async def stream(self, message: str, history=None, model=DEFAULT_MODEL):
    trimmed = _trim_history(history or [], MAX_HISTORY_MESSAGES)

    # BM25 search → top 3 docs (vulnerabilities or mitigations)
    kb_context = _build_kb_context(message)
    augmented  = (
        f"{message}\n\n---\n**Relevant knowledge base context:**\n\n{kb_context}"
        if kb_context else message
    )

    messages = [Message(role=h["role"], content=h["content"]) for h in trimmed]
    messages.append(Message(role="user", content=augmented))

    response = await self._router.generate(
        ModelRequest(model=model, messages=messages, system_prompt=SYSTEM_PROMPT,
                      metadata={"max_completion_tokens": MAX_TOKENS})
    )

    # Chunk and emit as SSE delta events
    for i in range(0, len(response.text), CHUNK_SIZE):
        yield SSEDeltaEvent(text=response.text[i : i + CHUNK_SIZE])
    yield SSEDoneEvent()

_build_kb_context searches both vulnerability and mitigation documents, formats each result with its name, description, impact level, tags, and a link to the relevant app page. Conversation history is trimmed to the last MAX_HISTORY_MESSAGES turns before each call to prevent token overflows.

Why SSE over WebSockets: chat responses flow in one direction only — server to client. SSE requires no handshake, no upgrade negotiation, and is natively supported by the browser's EventSource API. FastAPI's StreamingResponse with media_type="text/event-stream" is a two-line implementation.
Prompt Enhancer — Three-Stage Verified Pipeline Application

enhance_prompt_with_validation() orchestrates a three-stage pipeline that hardens a user's system prompt and verifies the result before returning it. If verification fails, the pipeline retries up to max_retries times (default 3), using the previous Stage 1 output as the new base to progressively improve each attempt.

Retry loop — app/enhancer/prompt_enhancer_service.py
for attempt in range(max_retries):   # default max_retries = 3

    # Stage 1 — LLM rewrites the prompt for clarity, preserving original intent
    improved_prompt = await improve_prompt_structure(original_prompt, provider_router, model_id)

    # Stage 2 — deterministic: prepend "# Security Guidelines" block from MITIGATION_REGISTRY
    enhanced_prompt = prepend_mitigations(improved_prompt, mitigation_ids)

    # Stage 3 — LLM verifies: intent preserved? all mitigations present? coherent?
    verification = await verify_enhancement(
        original_prompt, improved_prompt, enhanced_prompt, mitigation_ids, ...
    )

    if verification.get("verdict") == "PASS":
        return {
            "original_prompt":   original_prompt,
            "improved_prompt":   improved_prompt,
            "enhanced_prompt":   enhanced_prompt,
            "verification_data": verification,
            "attempts":          attempt + 1,
        }

raise EnhancementValidationError("Failed to validate after all retries")

Stage 2 is entirely deterministic: prepend_mitigations() iterates the selected mitigation IDs, reads each mitigation's prompt_message from MITIGATION_REGISTRY, and concatenates them under a # Security Guidelines header above a # System Instructions separator. No LLM call is made in this stage. Stage 3 sends the original, improved, and enhanced prompts to the model with a structured JSON response schema — the verdict field ("PASS" | "FAIL") drives the retry decision.

Why verified enhancement: a single-pass LLM rewrite may silently omit a mitigation or subtly alter the prompt's intended behaviour. Stage 3 catches these failures programmatically. In practice over 95% of enhancements pass on the first attempt; the retry loop exists for the rare cases of LLM drift.

6. Database

Persistence is provided by a single SQLite file (backend/app.db) accessed through SQLAlchemy's ORM. Three tables are used.

TablePurposeNotable columns
tests Stores saved test configurations that can be loaded and re-run model_json, environment_json, runner_json — domain objects serialised as JSON text
test_configurations Stores named presets (saved before execution) separately from executed tests Same JSON column structure as tests
prompt_enhancements Stores the full output of each enhancement run for review and comparison original_prompt, enhanced_prompt, enhancement_type

JSON column strategy: complex nested domain objects (ModelSpec, EnvironmentSpec, RunnerSpec) are stored as JSON text rather than normalised relational columns. On read, TestRepository._to_domain() reconstructs domain objects field-by-field with explicit type coercions (e.g. converting runner["type"] to a RunnerType enum). This keeps the SQL schema stable as the domain model evolves — new optional fields can be added without altering table columns.

7. End-to-End Workflows

Three complete use-case flows are traced below — from the user's first action in the browser to the final result on screen. Numbered steps make it easy to follow each stage.

Prompt-Runner Test

Use Case 1 — Prompt-Runner Test: A user configures a test (model, system prompt, mitigations), submits an adversarial prompt, and receives a risk-scored analysis showing whether the model was compromised.

The model call routes through Azure AI Foundry for platform models. For Garak runs the flow diverges at step 4: GarakRunner writes a temporary YAML config, invokes garak as a managed subprocess, and parses the JSONL hit report into a DEFCON-aggregated RunResult — no hybrid analyser is used.

Vulnerability & Mitigation Learning

Use Case 2 — Vulnerability & Mitigation Learning: A user browses the vulnerability catalogue, reads an attack detail page with an interactive demo, takes a knowledge quiz, then explores the mitigation library filtered by pipeline layer.
